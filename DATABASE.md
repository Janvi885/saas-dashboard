# Database Design — Firestore Schema

This document describes the Firestore data model for the SaaS Product Dashboard: collections, fields, indexes, security rules, and scaling considerations.

---

## Why Firestore (not SQL)

| Factor | Firestore fit for this project |
|--------|--------------------------------|
| **Firebase Auth integration** | Custom claims (`role`) align with Security Rules and Admin SDK writes |
| **Document model** | Products are self-contained documents with optional fields (`sku`, `stock`) |
| **Serverless / low ops** | No database server to manage; pairs with Vercel + Render deployment |
| **Real-time ready** | `onSnapshot` listeners can be added later without schema migration |
| **Demo scale** | Hundreds to low thousands of products; in-app pagination and aggregates are sufficient |

**When SQL would be better:** complex joins across orders/invoices, full-text search at scale, strict ACID multi-row transactions, or heavy reporting across normalized tables. This app models a single product catalog with RBAC — a document store is appropriate.

## Collections overview

| Collection | Path | Purpose |
|------------|------|---------|
| Products | `/products/{productId}` | Product catalog (CRUD) |
| Users | `/users/{userId}` | Profile mirror of Auth uid + role |
| Rate limits | `/rate_limits/{userId}` | AI description rate limiting |
| Meta | `/meta/seed` | Idempotent seed marker (internal) |

All client-facing CRUD goes through the **Express API + Admin SDK**. Firestore Security Rules are a second layer for direct client SDK access.

## `/products/{productId}`

Document ID is auto-generated by Firestore on create (`collection.doc()`).

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | `string` | Yes | Display name, 2–100 characters |
| `category` | `string` | Yes | One of: `Electronics`, `Clothing`, `Food`, `Software`, `Home`, `Books`, `Other` |
| `price` | `number` | Yes | USD amount stored as **number** (not string); validated 0–999,999.99 |
| `status` | `string` | Yes | `active` \| `inactive` |
| `description` | `string` | No | Optional marketing copy, max 500 chars |
| `sku` | `string` | No | Optional SKU, alphanumeric + dashes |
| `stock` | `number` | No | Optional inventory count, non-negative integer |
| `createdAt` | `Timestamp` | Yes | Set with `FieldValue.serverTimestamp()` on create |
| `updatedAt` | `Timestamp` | Yes | Set with `FieldValue.serverTimestamp()` on create/update |
| `createdBy` | `string` | Yes | Firebase Auth **uid** of the admin who created the product |

**API response shape:** The `id` field is the document ID (not stored inside the document body). The service maps `doc.id` → `Product.id` in JSON responses.

**Timestamp policy:** Production writes use `FieldValue.serverTimestamp()` only. The seed script uses the same for demo data.

**Price policy:** Zod validates `z.number()` on create/update; `mapDocToProduct` coerces reads to number defensively.

## `/users/{userId}`

Document ID equals Firebase Auth **uid**.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `uid` | `string` | Yes | Same as document ID |
| `email` | `string` | No | User email from Firebase Auth |
| `role` | `string` | Yes | `admin` \| `viewer` — mirror of custom claim |
| `createdAt` | `Timestamp` | No | Set on first profile write |
| `updatedAt` | `Timestamp` | Yes | Updated on role/profile changes |

**Source of truth for roles:** Firebase Auth **custom claims** (`request.auth.token.role`). The `/users` document is a readable mirror for the app and admin tooling.

---

## `/rate_limits/{userId}`

Document ID equals the requesting user's **uid**. Used by the AI description feature (Groq).

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `count` | `number` | Yes | Requests in the current window |
| `windowStart` | `Timestamp` | Yes | Start of the rolling 1-hour window |

**Access:** Written only by the backend (Admin SDK). Security Rules deny all client read/write.

**Limit:** 10 AI generations per user per hour.

## `/meta/seed`

Internal marker for the idempotent seed script.

| Field | Type | Description |
|-------|------|-------------|
| `seededAt` | `Timestamp` | When seed last completed |
| `version` | `number` | Seed schema version |

## Indexes

Firestore requires **composite indexes** when combining equality filters (`where`) with `orderBy` on a different field.

Defined in [`firestore.indexes.json`](./firestore.indexes.json). Deploy with:

```bash
firebase deploy --only firestore:indexes
```

### Indexed query patterns

| Query pattern | Index fields | Why |
|---------------|--------------|-----|
| Filter by category, sort by date | `category` + `createdAt` | Product list with category filter |
| Filter by status, sort by date | `status` + `createdAt` | Active/inactive tabs |
| Filter by category + status | `category` + `status` + `createdAt` | Combined filters |
| Filter + sort by name/price | `category`/`status` + `name`/`price` | User-selected sort columns |

**Single-field indexes** (e.g. `orderBy('createdAt')` with no filters) are created automatically by Firestore.

**Text search:** Firestore has no native `LIKE` / full-text search. When the `search` query param is set, the API applies filters at the database level then matches text **in memory** before paginating. For production search at scale, use Algolia, Typesense, or a `keywords` array field with `array-contains`.

## Pagination

The product list API supports `page`, `pageSize`, `sortBy`, and `sortOrder`.

| Scenario | Strategy |
|----------|----------|
| No text search | Firestore `where` + `orderBy` + `offset` + `limit`; total from `count()` aggregation |
| With text search | Filtered Firestore query, then in-memory search match + slice |

Default page size: **20** (max **100** via Zod validation).

This avoids loading the entire collection on every unfiltered list request.

**Note:** Analytics still reads all product price/status/category fields via a projected query (`select()`) for dashboard aggregates — acceptable at demo scale; would move to pre-aggregated counters or BigQuery at scale.

## Security rules

File: [`firestore.rules`](./firestore.rules)

| Collection | Admin | Viewer | Unauthenticated |
|------------|-------|--------|-----------------|
| `/products` | read, create, update, delete | read only | denied |
| `/users/{uid}` | read all, write | read own doc only | denied |
| `/rate_limits/{uid}` | denied | denied | denied |
| All other paths | denied | denied | denied |

Product writes from clients must pass `isValidProduct()` (name, price as number, status enum).

Deploy rules:

```bash
firebase deploy --only firestore:rules
```

## Multi-tenancy scaling path

Today the app is **single-tenant** (one shared catalog). To support organizations:

1. Add `organizationId: string` to every `/products/{productId}` document.
2. Set `organizationId` from the user's custom claim or `/users/{uid}.organizationId`.
3. Scope all queries: `.where('organizationId', '==', tenantId)`.
4. Update Security Rules:

   ```
   allow read: if isAdminOrViewer()
     && resource.data.organizationId == request.auth.token.organizationId;
   ```

5. Add composite indexes including `organizationId` as the leading field, e.g.:
   - `organizationId` + `category` + `createdAt`

No migration of existing documents is required for the demo; new tenants would get isolated collections via the `organizationId` field on insert.

```
Firebase Auth (uid, email, custom claims: role)
       │
       ├──► /users/{uid}          (profile mirror)
       ├──► /rate_limits/{uid}    (AI usage)
       └──► createdBy on /products/{id}
```
